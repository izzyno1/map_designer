# 单草稿地图标注器改造设计

日期：2026-04-03

## 目标

把当前“骑行路线地图标注后台”收敛为一个更轻的内部地图标注工具：

- 默认首页不再展示路线列表
- 首页只提供一个入口：`新建标注`
- 系统内只维护一份“当前标注稿”
- 标注对象只有两类：
  - 标注点
  - 赛段
- 赛段不再依附路线 geometry，也不依赖起止索引
- 用户完成标注后，可以保存当前稿并导出自定义 JSON，供另一套软件继续消费

## 非目标

这次改造不再保留以下产品语义：

- 路线列表
- 路线详情页
- 创建路线
- 路线 geometry 编辑
- 一键贴路
- 基于路线索引的赛段高亮
- 按 route_id 拆分的多路线管理

## 用户流

### 首页

首页为极简入口页：

- 标题说明当前工具是“地图标注器”
- 提供单个主按钮：`新建标注`
- 点击后进入唯一的标注编辑页

系统不再让用户选择路线，也不展示路线卡片列表。

### 编辑页

编辑页保留当前三栏后台布局，但语义重定义：

- 左侧：标注清单
  - 标注点列表
  - 赛段列表
- 中间：地图画布
- 右侧：当前选中对象的编辑面板

编辑页顶部保留工具操作区，动作改成面向标注：

- `新增标注点`
- `新增赛段`
- `保存标注`
- `导出 JSON`

## 数据模型

### 标注点

```json
{
  "id": "poi-1",
  "name": "补给点 A",
  "remark": "这里可以补水",
  "lat": 37.86,
  "lng": 112.54
}
```

字段说明：

- `id`：前端生成或后端分配的唯一标识
- `name`：标注点名称
- `remark`：备注
- `lat` / `lng`：地图位置

### 赛段

```json
{
  "id": "segment-1",
  "name": "赛段 A",
  "remark": "上坡测试段",
  "points": [
    { "lat": 37.87, "lng": 112.55 },
    { "lat": 37.88, "lng": 112.57 },
    { "lat": 37.89, "lng": 112.58 }
  ]
}
```

字段说明：

- `id`：唯一标识
- `name`：赛段名称
- `remark`：备注
- `points`：赛段折线点集，最少两个点

### 当前标注稿

```json
{
  "viewport": {
    "center": { "lat": 37.87, "lng": 112.55 },
    "zoom": 12
  },
  "pois": [],
  "segments": []
}
```

字段说明：

- `viewport`：当前地图中心点和缩放，用于重新打开时恢复视野
- `pois`：标注点集合
- `segments`：赛段集合

系统内只保存一份当前标注稿，不做多稿管理。

## 地图交互

### 普通模式

默认进入普通浏览模式：

- 可缩放
- 可拖动
- 可点击现有对象进行选中

### 新增标注点

点击 `新增标注点` 后：

- 页面进入“等待点位”状态
- 用户在地图上点击一次
- 立即创建一个标注点
- 自动选中该标注点
- 右侧打开标注点表单

### 新增赛段

点击 `新增赛段` 后：

- 页面进入“赛段绘制模式”
- 用户在地图上连续点击多个点
- 地图实时显示正在生成的折线
- 用户通过以下任一方式完成绘制：
  - 点击顶部“完成赛段”
  - 双击地图结束
- 若点数不足两个，不能完成
- 完成后自动选中新赛段，并在右侧显示赛段表单

### 选中与定位

- 点击左侧列表项，地图定位到对应对象
- 点击地图上的点或线，也会同步选中
- 右侧仅展示当前选中对象的编辑信息

## 编辑面板

### 标注点编辑

标注点只保留这些字段：

- 名称
- 经纬度（只读显示，第一版不做表单编辑）
- 备注

提供删除按钮，并在这轮做成可用功能。

### 赛段编辑

赛段只保留这些字段：

- 名称
- 点数统计（只读）
- 备注

第一版不在右侧直接逐点编辑赛段坐标，坐标修改以“删除后重画”为主。

提供删除按钮，并在这轮做成可用功能。

## 保存与导出

### 保存

保存对象从“路线及其附属数据”改为“当前唯一标注稿”。

后端接口调整为：

- `GET /api/v1/draft`
  - 获取当前标注稿
- `PUT /api/v1/draft`
  - 覆盖保存当前标注稿

保存采用整稿覆盖写入，不做局部 PATCH。

前端行为：

- 进入编辑页后先加载当前标注稿
- 若接口失败，则 fallback 到本地 mock 空稿
- 点击“保存标注”后，将整个当前稿发给后端

### 导出

后端提供：

- `GET /api/v1/draft/export`

导出结果为自定义 JSON，结构与当前标注稿一致，供外部软件消费。

## 前端改造边界

### 路由

建议路由改成：

- `/`
  - 首页，展示“新建标注”入口
- `/draft`
  - 标注编辑页

旧的 `/routes/:routeId` 入口可以移除，或短期内重定向到 `/draft`。

### 组件

建议保留现有三栏布局的基础设施，但做职责替换：

- 保留 `AppShell`
- 保留 `MapCanvas`，但移除路线 geometry 强依赖
- `AnnotationSidebar` 改为展示当前稿下的标注点和赛段
- `PoiEditorPanel` 保留并继续简化
- `SegmentEditorPanel` 改造成赛段元数据编辑 + 删除
- `RouteEditorPage` 改名并改造成 `MapAnnotationPage`
- 新增首页入口页，例如 `DraftEntryPage`

### 状态管理

新增单稿 hook，例如：

- `useDraftAnnotationData`

职责：

- 加载当前稿
- 管理保存状态
- 管理当前选中对象
- 管理创建模式：
  - 普通模式
  - 新增标注点模式
  - 新增赛段绘制模式
- 处理导出

旧的 `useRouteEditorData` 中与 `routeId`、`geometry`、贴路、路线详情相关的逻辑需要移除。

## 后端改造边界

后端从“多路线资源”改成“单标注稿资源”：

- 继续使用 Express + SQLite
- SQLite 可以先采用单表 JSON 存储，避免为单稿场景保留 route/poi/segment 关系模型

建议新增一个草稿存储模块，负责：

- 初始化默认空稿
- 读取当前稿
- 覆盖保存当前稿
- 生成导出 JSON

原有 route 相关接口可以先保留为兼容代码，但前端不再使用。

## 错误处理

- 获取草稿失败：前端提示“加载失败，已进入本地演示稿”
- 保存失败：提示“保存失败，请稍后重试”
- 导出失败：提示“导出失败，请确认后端服务已启动”
- 赛段完成时点数不足：提示“赛段至少需要两个点”

## 测试范围

### 前端

- 首页展示“新建标注”按钮
- 点击后进入 `/draft`
- 空稿可正常渲染
- 新增标注点后，右侧能编辑名称和备注
- 新增赛段后，地图显示折线，右侧能编辑名称和备注
- 删除标注点/赛段后，列表和地图同步更新
- 保存调用 `PUT /api/v1/draft`
- 导出调用 `GET /api/v1/draft/export`

### 后端

- `GET /api/v1/draft` 返回默认空稿
- `PUT /api/v1/draft` 可写入合法草稿
- 非法草稿返回 400
- `GET /api/v1/draft/export` 返回 JSON 文件响应

## 实施原则

- 优先复用现有地图、面板和三栏布局
- 明确砍掉路线语义，不做“兼容式半保留”
- 第一版让赛段以“绘制 + 删除重画”为主，不做复杂节点编辑
- 保持前端接口失败自动 fallback，确保本地和演示环境都能跑
