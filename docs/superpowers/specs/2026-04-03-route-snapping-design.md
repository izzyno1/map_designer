# 骑行路线道路贴合优化设计

## 背景

当前路线 geometry 直接按坐标点渲染为 Leaflet `Polyline`，相邻点之间使用直线连接。  
这会让路线在地图上看起来像“折线草图”，不够接近真实骑行道路。

本轮优化目标是为现有路线编辑器增加“一键贴路”能力：基于当前整条路线的坐标点，调用在线路线服务生成沿道路的完整折线，并让运营人员在保存前先预览结果。

## 目标

- 为路线编辑器增加整条路线的一键道路贴合能力
- 保持现有 geometry 编辑和保存流程不变
- 贴路结果先预览，不自动写数据库
- 同时在顶部工具栏和右侧 `路线坐标` 面板提供入口
- 后端统一封装第三方路线服务，前端不直接依赖第三方接口

## 非目标

- 不实现局部路段贴合
- 不实现贴路前后双路线对比视图
- 不实现自动保存
- 不接入自建 OSM 路由服务
- 不为贴路结果提供 mock 伪造数据

## 方案选择

本轮采用：后端代理在线 OSRM 路线服务。

原因：

- 前端只调用自有 `/api`，结构更清晰
- 未来切换到自建路由服务时，前端可保持不变
- 错误处理、限流、日志等能力集中在后端

不采用前端直连第三方服务，也不采用本轮直接自建路由引擎。

## 用户流程

1. 运营进入某条路线编辑页
2. 点击顶部工具栏或右侧 `路线坐标` 面板中的 `一键贴路`
3. 页面进入贴路中状态，按钮禁用
4. 后端基于当前整条 geometry 调用 OSRM 骑行路线服务
5. 返回新的沿道路 geometry
6. 前端只更新当前页面的 geometry draft，并立即在地图上预览
7. 页面提示：`已生成道路贴合结果，请确认后保存`
8. 用户满意后，再点击已有的 `保存路线坐标`
9. geometry 按原有流程写入数据库

如果贴路失败，则保留原 geometry 不变，并显示错误提示。

## 接口设计

新增后端接口：

- `POST /api/v1/routes/:routeId/snap-geometry`

请求体：

```json
{
  "geometry": {
    "type": "LineString",
    "coordinates": [
      { "lat": 37.87, "lng": 112.56, "distanceKm": 0 },
      { "lat": 37.88, "lng": 112.57, "distanceKm": 1.2 }
    ]
  }
}
```

成功响应：

```json
{
  "geometry": {
    "type": "LineString",
    "coordinates": [
      { "lat": 37.8701, "lng": 112.5602, "distanceKm": 0 },
      { "lat": 37.8706, "lng": 112.5608, "distanceKm": 0.1 }
    ]
  }
}
```

错误响应：

- `400`：geometry 无效或点数不足
- `404`：route 不存在
- `502`：上游道路贴合服务失败或返回异常结果

## 后端设计

### 新模块

新增独立服务模块，例如：

- `server/services/snap-geometry.js`

职责：

- 校验输入 geometry
- 将现有坐标转换为 OSRM 所需的 `lng,lat;lng,lat...` 字符串
- 调用 OSRM `route` 接口，profile 使用 `bike`
- 请求参数固定使用 `overview=full` 与 `geometries=geojson`
- 从返回结果中提取路线折线
- 转回项目现有 `RouteGeometry`
- 重新计算 `distanceKm`

### 环境变量

新增可选环境变量：

- `OSRM_BASE_URL`

默认值可指向公开服务，例如：

- `https://router.project-osrm.org`

### 结果处理

后端只返回贴路后的 geometry，不直接改数据库。  
数据库写入仍然复用已有：

- `PATCH /api/v1/routes/:routeId/geometry`

## 前端设计

### 新 API

在前端 `api/routes.ts` 中新增：

- `snapRouteGeometry(routeId, geometry)`

只调用自有后端接口，不直接请求 OSRM。

### 页面交互

入口同时放两处：

- 页面顶部工具栏
- 右侧 `路线坐标` 面板

两处都触发同一个行为：

- 请求当前 geometry draft
- 成功后更新 geometry draft
- 不自动保存

### 状态

新增单独的贴路中状态，例如：

- `snapping: boolean`

行为：

- 贴路中时两个入口都禁用
- 页面显示进行中反馈
- 成功提示：`已生成道路贴合结果，请确认后保存`
- 失败提示：`道路贴合失败，请稍后重试`

### 数据更新范围

贴路成功后只更新：

- 当前页面的 geometry draft
- 依赖 geometry draft 的地图折线与右侧坐标列表

不直接更新：

- 数据库
- 路线详情原始数据

## 错误处理

- geometry 不存在或点数少于 2 个时，不允许贴路
- OSRM 不可达时，后端返回 `502`
- OSRM 返回空路线或结构异常时，后端返回 `502`
- 前端收到失败响应时，保留原 geometry，不覆盖当前编辑状态

本轮不为贴路功能提供 mock 数据。  
原因是贴路结果高度依赖真实路径服务，伪造结果会误导运营判断。

## 测试范围

### 后端

- 有效 geometry 可成功调用贴路服务并返回新的 geometry
- 点数不足时返回 `400`
- route 不存在时返回 `404`
- 上游失败时返回 `502`

### 前端

- 点击 `一键贴路` 会调用新接口
- 成功后更新 geometry draft
- 成功后显示确认保存提示
- 失败后显示错误提示
- 无 geometry 或点数不足时按钮禁用

## 风险与限制

- 依赖在线 OSRM 服务，受网络和上游稳定性影响
- 公开服务可能存在限流，不适合极高频批量调用
- 贴路结果质量取决于当前输入点是否足够代表真实路线

## 后续演进

后续如果需要更强控制力，可保持前端不变，仅把后端的 OSRM 基础地址切换为自建路由服务。  
如果后续出现高频局部修线需求，再单独设计“局部重贴路”能力。
